# SPEC: geiger-tube firmware — 乱数の管・第二源泉 (ESP32)

## 目的

CAJOE / RadiationD v1.1系ガイガーカウンターのパルス出力から崩壊イベントの
到着時刻を取得し、間隔からエントロピーを抽出して kuda (Cloudflare Worker) の
POST /ingest へ送る常駐ファームウェア。7/25の四谷ラボイベントで
「組み上がったガイガー基板を差すだけで管に繋がる」状態にすることがゴール。

## ハードウェア前提

- ボード: Freenove ESP32-WROOM-32E (Arduino framework / PlatformIO推奨)
- 信号入力: GPIO 4 (基板のVIN=パルス出力。負論理、通常High→検出時Low)
- 給電: ESP32のUSB-Cから。ガイガー基板へは5Vピンから供給
- オンボードLED: パルス検出時に短く点滅(視覚確認用)

## 設定の分離 (必須)

`config.h` (gitignore対象) に以下を分離。`config.example.h` をリポジトリに置く:
- WIFI_SSID / WIFI_PASSWORD
- PIPE_URL (例 https://kuda.kojiran.workers.dev)
- INGEST_TOKEN
- GPIO_PULSE (default 4), 各種閾値

INGEST_TOKENをシリアルログに出力しないこと。

## 処理パイプライン

1. **割り込み**: GPIO立ち下がりエッジでISR起動。ISRは micros() を
   リングバッファ(容量512イベント)に積むだけ。処理はループ側で行う。
2. **デッドタイム除去**: 前イベントから 300µs 未満のパルスは破棄
   (GM管のデッドタイム+チャタリング対策)。
3. **間隔計算**: 隣接イベントの到着間隔 Δt (µs, uint32) を得る。
4. **抽出**: Δt の下位8bitを収集バッファに追加。
5. **白色化**: 収集バッファが256イベント分(256バイト)溜まったら
   SHA-256でハッシュし、出力32バイトを送信キューへ。
   → エントロピー計上は保守的に「1イベント=1bit」(256イベント→256bit=32B)。
6. **送信**: 送信キューが64バイト以上、または最終送信から10分経過かつ
   32バイト以上で、POST /ingest:
   `{"bytes": "<base64>", "source": "geiger"}` + `Authorization: Bearer <token>`
   - HTTPS必須。ルートCA検証を実装(不可能ならsetInsecureにTODOコメント明記)
   - 200が返るまでキューをクリアしない(再送で二重投入しない=200時のみクリア)
   - リトライは指数バックオフ(30s→60s→120s→…上限10分)

## 規律 (変更禁止)

- **疑似乱数で埋めない**: イベントが少なくても、乱数を捏造して送らない。
  溜まった分だけ送る。管が細い日は細いまま。
- **二重投入禁止**: 同じバイト列を二度POSTしない(上記200時クリアで担保)。
- **バッファ満杯時**: 送信不能でキューが上限(4KiB)に達したら、新規蓄積を
  停止してシリアルに警告。古い粒の破棄も新しい粒の上書きもしない。

## 可観測性 (シリアル 115200)

10秒ごとに1行でステータス出力:
`CPM=23 events_total=1234 queue=96B wifi=OK last_post=200(45s ago)`
- 初回起動時: WiFi接続結果、kudaへの疎通確認(GET /status)の結果を表示
- イベント当日はこの画面をプロジェクタに映す想定。人が読める簡潔さを保つ

## テストモード

- `config.h` の `TEST_MODE=1` で、GPIO割り込みの代わりに擬似ポアソン過程
  (平均25CPM、乱数はesp_random()でよい※テスト専用)でパイプラインを駆動。
  ガイガー未到着でも配線以外の全経路を検証できるようにする。
  TEST_MODE時は source="test" で送信し、本番の geiger# と混ざらないこと。

## 受け入れ条件

1. TEST_MODEでkudaの /status の pool_remaining が増える
2. 実パルス(タクトスイッチ手動でも可)でCPM表示とLED点滅が動く
3. WiFi断→復帰で蓄積分が送信され、二重投入がない
4. 電源断→再起動で正常に再開する(蓄積分の消失は許容)
5. INGEST_TOKENがログ・リポジトリに露出しない

## 非目標

- OTA更新、Web UI、LCD表示 (第二弾イベントの領分)
- カーネル/dev/randomへの供給 (tubedの領分)
- 線量計算 (µSv/h換算はこのプロジェクトの目的ではない)
